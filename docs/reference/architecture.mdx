## Scope

v0.1.0 ships exactly three concerns:

1. A **namespace convention** — what a Python distribution (and the
   image that bundles it) must contain to be consumable by Agentix.
2. A **runtime server** — one Python process per sandbox that loads
   each installed namespace class on first call and exposes typed
   remote dispatch + sandbox I/O.
3. A **Docker deployment** — assembles sandboxes from a runtime image
   plus one or more namespace images and starts the runtime.

See [ROADMAP.md](https://github.com/Agentiix/Agentix/blob/master/ROADMAP.md)
for what comes later.

## Components

```text
┌─ Host (orchestrator) ─────────────────────────────────────────┐
│  RuntimeClient                                                 │
│    • run / upload / download           (runtime built-ins)     │
│    • namespaces                        (introspection)         │
│    • remote(fn, *args, **kwargs)       (typed dispatch)        │
└──────────────────────────────┬─────────────────────────────────┘
                               │ HTTP (POST /_remote)
┌─ Sandbox ────────────────────▼─────────────────────────────────┐
│                                                                 │
│  agentix-server (single Python process)                         │
│    built-in I/O:                                                │
│      GET  /health                                               │
│      POST /exec     (SSE or JSON)                               │
│      POST /upload                                               │
│      GET  /download                                             │
│      GET  /namespaces                                           │
│    typed dispatch:                                              │
│      POST /_remote   { package, method, args, kwargs }          │
│      /socket.io/     streams, bidi, log fan-out                 │
│                                                                 │
│  Registry: package → class (in-process, no subprocesses)        │
│    populated at startup from                                    │
│    importlib.metadata.entry_points(group="agentix.namespace");  │
│    ep.load() deferred to first /_remote call per namespace.     │
│                                                                 │
│  /nix/store — only present if a namespace shipped default.nix;  │
│  baked into the bundle image at build time                      │
└─────────────────────────────────────────────────────────────────┘
```

The runtime walks `importlib.metadata.entry_points` at startup and
registers each pending entry; first dispatch triggers the actual import.
Namespaces are fixed for the sandbox's lifetime — change the set by
rebuilding the bundle image and recreating the sandbox.

## Namespace convention

A namespace is a Python distribution that contributes a class under
`agentix.<short>` and declares one entry point under `agentix.namespace`.
Layout:

```text
my-namespace/
├── pyproject.toml            # name = "agentix-<short>", entry-point block
├── src/agentix/<short>/      # PEP 420 namespace package
│   └── __init__.py           # class <Name>(Namespace) with @staticmethod methods
└── default.nix               # OPTIONAL — only when the namespace needs
                              #   system deps (claude CLI, git, libffi, …)
```

Routing is by Python module path. After bundling, `from agentix.<short> import Cls`
on the caller side and `c.remote(Cls.method, ...)` is all the contract.
Two distributions registering the same entry-point name raise
`PluginConflictError` on first lookup.

See [Namespace protocol](/reference/namespace-protocol) for the full ABI.

## Build pipeline

`agentix build <specs...> -o <tag>` produces a single deploy-ready
bundle image:

1. **Stage 1 (optional Nix):** if any spec ships `default.nix`, a
   `nixos/nix` builder stage runs first. Each spec's derivation is
   built; the full closure is copied to `/export/nix/store`. Bundles
   with no system deps anywhere skip this stage entirely.
2. **Stage 2 (per-namespace venvs):** `FROM agentix/runtime:<version>`
   (which is Python 3.11-slim + framework venv at `/venvs/runtime/` +
   uv + the framework wheel stashed at `/agentix-wheel/`). If stage 1
   ran, `COPY --from=sys-builder /export/nix /nix`. Then **for each
   spec** create its own venv via `uv venv /venvs/<short>/` and
   pip-install the namespace + the framework wheel into it.
3. **Output:** one image carrying the framework + one venv per
   namespace + any system deps. The sandbox runs it directly.

The runtime image (`agentix/runtime:<version>`) is built once via
`docker build -f primitives/_template/Dockerfile .` from the repo root.

## Sandbox layout

```text
/                                  ← bundle image rootfs
├── venvs/
│   ├── runtime/                   ← framework venv; ENTRYPOINT here
│   │   ├── bin/agentix-server     ← the multiplexer process
│   │   └── lib/python3.11/site-packages/agentix/...
│   ├── bash/                      ← one venv per namespace
│   │   ├── bin/python             ← worker interpreter for agentix.bash
│   │   └── lib/python3.11/site-packages/agentix/bash/...
│   └── files/
│       ├── bin/python
│       └── lib/python3.11/site-packages/agentix/files/...
├── agentix-wheel/                 ← framework wheel; reused at build time
└── nix/                           ← optional, only if a namespace had
    └── store/                       default.nix
        ├── <hash>-claude-*/bin/claude
        └── <hash>-git-*/bin/git
```

The sandbox container's `ENTRYPOINT` is `/venvs/runtime/bin/agentix-server`.
It binds to `AGENTIX_BIND_PORT` and starts the multiplexer; namespace
workers spawn lazily on first dispatch using each namespace's
`/venvs/<short>/bin/python`.

## Environment & PATH policy

Rules at every `Bash.run` invocation:

1. **Strip Nix-host-only env vars** — `LD_LIBRARY_PATH`, `LD_PRELOAD`,
   `PYTHONPATH`, `PYTHONHOME`, `LOCALE_ARCHIVE`, `FONTCONFIG_*`,
   `SSL_CERT_FILE`, anything prefixed `NIX_`.
2. **PATH defaults to** `/usr/local/bin:/usr/bin:/bin`.
3. **Namespace native tools** ship at content-addressed
   `/nix/store/<hash>/bin/<tool>` paths and are invoked by their
   absolute path from Python impls.
4. **Namespace Python impls run in the runtime's interpreter** —
   they invoke native tools via `subprocess` with `/nix/store` paths,
   which resolve directly since the store is baked into the image.

## Deployment (Docker)

Sandbox create reduces to one `docker run`:

```bash
docker run -d \
  --name <sandbox-id> \
  --network host \
  -e AGENTIX_BIND_PORT=<port> \
  <bundle-image>
```

The deployment picks a free host port, passes it via env, and
health-checks `/health` on it before returning. No volume-populate
dance, no mount-and-merge, no custom entrypoint — the bundle image
carries everything.

## Design decisions

- **Subprocess per namespace.** Each namespace runs in its own venv's
  Python interpreter, isolated from every other namespace's deps. The
  runtime process is the multiplexer — it spawns workers lazily,
  forwards RPC frames over stdin/stdout, and routes trace events back
  to the Socket.IO channel.
- **Entry-point discovery across venvs** — the multiplexer walks
  `/venvs/*/lib/python*/site-packages` for `agentix.namespace` entry
  points. No `manifest.json`, no per-framework decorator.
- **The class is the API spec** — `@staticmethod` methods on a
  `Namespace` subclass carry both the type signature and the real
  implementation. The host imports the class for typing; the body
  runs inside the worker subprocess.
- **Static namespace set per sandbox** — change the set by rebuilding
  the bundle and recreating the sandbox.
- **Built-in sandbox I/O on the runtime** — run / upload / download
  always available, independent of any namespace.
- **Nix is opt-in.** Pure-Python namespaces ship no `default.nix` and
  the build skips Nix entirely — `FROM python:slim` end-to-end, seconds
  to build instead of minutes.

## Out of scope (v0.1.0)

- Bearer-token auth on the runtime (sandbox-level trust assumed).
- Higher-level interfaces for agents / datasets / benchmarks — see
  [ROADMAP.md](https://github.com/Agentiix/Agentix/blob/master/ROADMAP.md).
