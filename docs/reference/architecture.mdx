## Components

```text
┌─ Host (orchestrator) ─────────────────────────────────────────┐
│  RuntimeClient                                                 │
│    • run / upload / download           (runtime built-ins)     │
│    • namespaces                        (introspection)         │
│    • remote(fn, *args, **kwargs)       (typed dispatch)        │
└──────────────────────────────┬─────────────────────────────────┘
                               │ HTTP (POST /_remote)
┌─ Sandbox ────────────────────▼─────────────────────────────────┐
│                                                                 │
│  agentix-server (multiplexer process)                           │
│    built-in I/O:                                                │
│      GET  /health                                               │
│      POST /exec     (SSE or JSON)                               │
│      POST /upload                                               │
│      GET  /download                                             │
│      GET  /namespaces                                           │
│    typed dispatch:                                              │
│      POST /_remote   { package, method, args, kwargs }          │
│      /socket.io/     streams, bidi, log fan-out                 │
│                                                                 │
│  Workers: one subprocess per namespace, spawned lazily on       │
│  first dispatch using the namespace's venv interpreter.         │
│                                                                 │
│  /nix/store — only present if a namespace shipped default.nix.  │
└─────────────────────────────────────────────────────────────────┘
```

The runtime walks `importlib.metadata.entry_points(group="agentix.namespace")`
at startup. First dispatch triggers worker spawn for the target
namespace. Namespaces are fixed for the sandbox's lifetime — change
the set by rebuilding the bundle and recreating the sandbox.

## Namespace convention

```text
my-namespace/
├── pyproject.toml            # name = "agentix-<short>", entry-point block
├── src/agentix/<short>/      # PEP 420 namespace package
│   └── __init__.py           # public async functions
└── default.nix               # OPTIONAL — system deps only
```

Routing is by Python module path: `c.remote(bash.run, ...)` reads
`bash.run.__module__` as the routing key. Two distributions
registering the same entry-point name raise `PluginConflictError`.

See [Namespace protocol](/reference/namespace-protocol) for the full
ABI.

## Build pipeline

`agentix build [path]` packages one project root into a deploy-ready
bundle. Plugins arrive transitively via pip from the project's
`[project].dependencies`; the CLI never enumerates them.

1. **Optional Nix stage:** if the project ships `default.nix`, a
   `nixos/nix` builder stage runs first; the derivation closure is
   copied to `/export/nix/store`. Projects with no system deps skip
   this stage.
2. **One pip install:** `FROM agentix/runtime:<version>` (Python
   3.11-slim + framework venv at `/nix/runtime/`). One
   `pip install /src/project` resolves the user's project + every
   declared `agentix-*` plugin into the same venv. If the project
   shipped `default.nix`, its derivation's `bin/*` is symlinked into
   `/nix/runtime/bin/`.

The runtime image (`agentix/runtime:<version>`) is built once via
`docker build -f ../Agentix-Runtime-Basic/runtime/Dockerfile .`.

## Sandbox layout

```text
/                                  ← bundle image rootfs
├── nix/
│   ├── runtime/                   ← framework + every plugin + user code
│   │   ├── bin/agentix-server     ← ENTRYPOINT; the multiplexer process
│   │   ├── bin/python             ← every worker's interpreter
│   │   ├── bin/claude             ← symlink → /nix/store/<hash>/bin/claude
│   │   └── lib/python3.11/site-packages/
│   │       ├── agentix/...        ← framework
│   │       ├── agentix/bash/      ← from agentix-runtime-basic
│   │       ├── agentix/claude_code/ ← from agentix-claude-code
│   │       └── my_project/        ← user's project
│   └── store/                     ← Nix store (only if default.nix shipped)
│       ├── <hash>-claude-*/bin/claude
│       └── <hash>-git-*/bin/git
```

ENTRYPOINT is `/nix/runtime/bin/agentix-server`, binding
`AGENTIX_BIND_PORT`. One worker subprocess per dispatched module
spawns from `/nix/runtime/bin/python`.

## Environment & PATH policy

Subprocess invocations:

1. **Strip Nix-host-only env vars** — `LD_LIBRARY_PATH`, `LD_PRELOAD`,
   `PYTHONPATH`, `PYTHONHOME`, `LOCALE_ARCHIVE`, `FONTCONFIG_*`,
   `SSL_CERT_FILE`, anything prefixed `NIX_`.
2. **PATH defaults to** `/usr/local/bin:/usr/bin:/bin`, with
   `/nix/runtime/bin/` prepended for every worker so symlinked Nix
   binaries (`claude`, `git`, …) resolve by bare name.
3. **Nix-store paths** (`/nix/store/<hash>/bin/<tool>`) are stable
   across the bundle's lifetime — content-addressed.

## Deployment (Docker)

```bash
docker run -d \
  --name <sandbox-id> \
  --network host \
  -e AGENTIX_BIND_PORT=<port> \
  <bundle-image>
```

The deployment picks a free host port, passes it via env, and
health-checks `/health` before returning. The bundle image carries
everything — no volume populate, no mount-and-merge.
